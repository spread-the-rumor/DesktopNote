# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this app is

An Electron desktop app that uses the **Recall AI Desktop SDK** (`@recallai/desktop-sdk`) to detect meetings (Google Meet in Chrome/Edge on Windows), record them, and retrieve transcripts by **polling** the Recall API after the recording ends (no inbound webhook / public endpoint required). Recording can also be started **manually** with a **"Huddle"** button (captures the screen + desktop/mic audio without a detected meeting). When a recording completes, it generates a structured **AI summary** of the transcript, **persists it as a meeting note** to a local JSON file, and presents it in a **notepad-style UI** — a sidebar of past meetings (with a **Trash** section at the bottom: deletes are soft, recoverable, and can be restored or permanently purged) plus an editor with three toggle views: the editable **Summary** (refinable before sending out), the read-only **Transcript**, and an **AI Chat** that answers questions grounded in that meeting's transcript. From the Summary view the note can be sent to two destinations via a **"Send to" Slack | GetOverview toggle**: **Slack** (channel or DM) and **GetOverview** (the user's internal project-management tool — submit the summary/transcript, or AI-extract action items into editable rows and create tasks in a project). Built with Electron Forge + webpack.

### Running the full pipeline locally

The full pipeline runs entirely locally — no public reachability, ngrok, or webhook config needed:
1. `npm start` — the in-process backend logs `Backend running on http://localhost:3100`.
2. The app must stay open after a call ends (or after pressing **Stop** on a Huddle) — Recall processes the recording server-side for minutes after the call, and the app **polls** for the finished media on the SDK's `recording-ended` event (see "completion" below). The same applies to the manual Huddle button, since it uses the identical pipeline.

> **No public endpoint / ngrok required.** Completion is **poll-based, not webhook-based.** The app no longer exposes a `/webhook` route; instead `main.js` captures the `desktop_sdk_upload` `id` at record-start and, on `recording-ended`, calls `backend.processCompletedUpload(uploadId)`, which polls `GET /api/v1/recording/?desktop_sdk_upload_id=<id>` (outbound only) until the media is ready. ngrok and the Recall dashboard webhook config are no longer used.

`.env` (gitignored) must define `RECALL_API_KEY` and `RECALL_API_URL` (see `.env.example`). The backend `process.exit(1)`s if the key is missing. The Slack **Bot User OAuth Token** (read as `Bot_User_OAuth_Token`, with `SLACK_BOT_TOKEN`/`SLACK_BOT_USER_OAUTH_TOKEN` fallbacks) is optional — it powers the "Send to Slack" feature (channel/person dropdown + send); if unset, listing and sending return a clear "not configured" error instead of failing. Required bot scopes: `chat:write`, `channels:read`, `groups:read` (private channels), `channels:join` (auto-join public channels), `users:read` (the People list), `im:write` (open DMs). The old `SLACK_WEBHOOK_URL` is **deprecated and unused** — the webhook (single-channel) was fully replaced by the bot-token Web API. The AI summary needs a requesty.ai key, read as `REQUESTY_API_KEY` with a fallback to `RECALLAI_REQUEST_KEY` (the name it's stored under in this project's `.env`); if neither is set, summaries (and the action-item extraction that reuses the same key) degrade to a readable "unavailable" string rather than crashing. The **GetOverview** integration is optional too: base URL + token are read as `GetOverview_BASE_URL` and `GetOverview_Access_Token` (exact names; trailing slash on the URL is stripped). The token is a Personal Access Token (`go_live_…`) sent as `Authorization: Bearer <token>`. If either is unset, all GetOverview controls report a clear "not configured" state instead of failing, and the rest of the app is unaffected.

## Architecture

Three processes cooperate; data flows in a loop **renderer ← main ← in-process backend ← Recall**.

- **`main.js` (Electron main process)** — Orchestrator. On `app.whenReady` it: starts the backend in-process (`backend.start(3100)`), points the history store at `app.getPath('userData')` (`store.init`), registers `setRecordingCompleteHandler` to **persist then forward** each completed meeting, registers **all** SDK event listeners, then calls `RecallAiSdk.init()` **last**. The complete handler calls `buildMeeting(payload)` (derives the date/title, seeds the editable `content` from the AI summary, and seeds an empty `chat: []`), `store.upsertMeeting`s it, and forwards the *saved meeting* on `recording-complete`. Also hosts the history IPC handlers (`list-meetings`, `update-meeting`, `delete-meeting`) **plus the trash handlers** (`list-trash`, `restore-meeting`, `delete-meeting-permanent` → the matching `store.*`), the chat handler (`ask-meeting` → `backend.askMeeting`), the manual-recording handlers (`start-huddle` / `stop-huddle`), the Slack handlers (`send-to-slack`, `list-slack-channels` → `backend.listSlackChannels`, `list-slack-users` → `backend.listSlackUsers`), the GetOverview handlers (`list-getoverview-projects`, `create-getoverview-task`, `send-getoverview-transcript`, `extract-action-items` → the matching `backend.*` — all thin pass-throughs like the Slack ones), plus `save-transcript` / `save-recording`. On `meeting-detected` it asks the backend for an upload token and calls `RecallAiSdk.startRecording`. **`start-huddle`** is the manual analogue: it calls `RecallAiSdk.prepareDesktopAudioRecording()` to get a desktop-audio `windowId` (no detected meeting needed), fetches an upload token from the **same** `create_sdk_recording` route, calls `startRecording`, and stores the id in a module-level `huddleWindowId`; **`stop-huddle`** calls `RecallAiSdk.stopRecording({ windowId })` and clears it. `acquirePermissionsOnStartup` includes `'system-audio'` (alongside accessibility/microphone/screen-capture) so desktop-audio capture is permitted. On quit it shuts the SDK down and closes the backend server.

- **`store.js` (required by main.js)** — Local meeting-history persistence: a single `meetings.json` in Electron's `userData` dir, shape `{ meetings: [ <meeting> ] }`. A meeting is `{ id (=recordingId), recordingId, title, date, content, summary, transcript, videoUrl, chat, createdAt, updatedAt }` — `content` is the editable note text (seeded from `summary`), `summary` keeps the original AI output, `chat` is the persisted Q&A thread (`[{ role:'user'|'assistant', content }]`). A trashed meeting additionally carries a `deletedAt` timestamp; restoring clears it. Exports `init / loadMeetings / loadTrash / upsertMeeting / updateMeeting / deleteMeeting / restoreMeeting / permanentlyDeleteMeeting`. **Trash is a soft-delete:** `deleteMeeting(id)` stamps `deletedAt` (it no longer removes the row), `restoreMeeting(id)` clears it, and `permanentlyDeleteMeeting(id)` is the old hard-delete (the only irreversible path). `loadMeetings` returns **live** notes only (no `deletedAt`), `loadTrash` returns **trashed** notes only (newest-trashed first); both are derived from a private `loadAll` raw read. `upsertMeeting`/`updateMeeting` also read via `loadAll`, so trashed rows survive every write. `updateMeeting(id, patch)` merges arbitrary patches, so the renderer persists summary edits, title edits, **and the chat thread** through the same path. Reads are forgiving (missing/corrupt file → `{ meetings: [] }`, never throws); writes serialize through a single in-flight promise chain (temp-file + rename) so the renderer's debounced auto-saves can't interleave/corrupt the file. No third-party dep.

- **`server.js` (Express backend, required in-process by main.js)** — Bridges the app to the Recall API. One route: `POST /api/create_sdk_recording` (creates an SDK upload with `video_mixed_mp4` + a transcription provider and returns the upload token **and `id`** — reused unchanged by **both** the auto-detect and the manual Huddle paths). Completion is **poll-based, not a webhook**: it exports `processCompletedUpload(uploadId)`, which `main.js` calls on `recording-ended`. That function polls `fetchRecordingByUploadId` (`GET /api/v1/recording/?desktop_sdk_upload_id=<id>`, outbound only, ~20 attempts with capped backoff over ~3.5 min) until the recording's transcript + mixed-video media is ready, downloads the transcript, calls `summarizeTranscript` to generate the AI summary, dedups by the resolved recording id (`handledRecordings` Set), and notifies main via the registered handler with `{ recordingId, videoUrl, transcriptUrl, transcript, summary }` — it does **not** write anything to disk. (The old `POST /webhook|/webhook/recall` route and the ngrok requirement are gone.) Also exports the **Slack Web API** helpers — `sendToSlack` (posts the note text to a channel via `chat.postMessage`, or to a person as a DM via `conversations.open` then `chat.postMessage` when `isDm` is set; auto-joins public channels on `not_in_channel`), `listSlackChannels` (`conversations.list`, public + private, cursor-paginated), and `listSlackUsers` (`users.list`, real humans only — skips bots/apps/deactivated/Slackbot) — all keyed off the bot token and **never throwing** (they return `{ ok, … }` | `{ ok:false, error }`, surfacing Slack's logical errors verbatim). Also exports `formatTranscript` (reused by main.js for manual transcript saves) and `askMeeting` (thin wrapper over `chat.js`'s `answerQuestion`, returns `{ ok, answer }`). Also exports the **GetOverview Web API** helpers (Bearer-auth, base URL from `.env`, all token-guarded and **never throwing** — same `{ ok, … }` | `{ ok:false, error }` contract as Slack, surfacing GetOverview's logical errors verbatim): a generic `getoverviewApi(method, pathname, body)` wrapper, `listGetOverviewProjects` (`GET /api/v1/projects` → `{ id, name, status, url }`, sorted by name; the renderer groups by `status`), `createGetOverviewTask` (`POST /api/v1/projects/{projectId}/tasks` with `title`/`assignee`/`dueDate`/`description`, empty optionals omitted), `sendTranscriptToGetOverview` (`POST /api/v1/transcripts` or the project-scoped `/api/v1/projects/{projectId}/transcripts`; trims `transcript` to GetOverview's 700 KB limit; returns `{ ok, jobId, status }` — the 202 result is not polled back), and `extractActionItems` (thin wrapper over `extractActionItems.js`, returns `{ ok, items, note }`). Also runs standalone via `node server.js`.

- **`summarize.js` (required by server.js)** — Exports `summarizeTranscript(transcript, participants = [], { stream = true, progressCallback })`, which calls the OpenAI-compatible requesty.ai chat-completions API (`https://router.requesty.ai/v1`, model `google/gemma-4-31b-it`) to produce a fixed-structure Markdown summary (`# Participants` / `# Summary` / `# Action Items`). Accepts both the documented `[{ speaker, text, timestamp }]` shape and Recall's native segment shape (normalizes both). Supports streaming (parses the SSE stream, fires `progressCallback(currentText)` per chunk) and non-streaming modes. **Never throws** — returns a human-readable `"Summary unavailable — …"` string on any failure so the UI can show it as-is.

- **`chat.js` (required by server.js)** — Exports `answerQuestion(transcript, question, history = [])`, the per-meeting AI chat. Same requesty.ai plumbing as `summarize.js` (`BASE_URL`/`MODEL`, `REQUESTY_API_KEY → RECALLAI_REQUEST_KEY` fallback, identical `Speaker: text` transcript rendering). System prompt answers **only** from the transcript (says so when the answer isn't there); sends the **full untruncated transcript** plus the prior `history` turns for follow-ups; non-streaming. **Never throws** — returns a `"Chat unavailable — …"` string on failure. `summarize.js` is left untouched; this is a sibling module so the summary path is unaffected.

- **`extractActionItems.js` (required by server.js)** — Exports `extractActionItems(transcript, summary = '')`, used to pre-fill the "Create tasks in GetOverview" editor. Same requesty.ai plumbing as `chat.js`/`summarize.js` (`BASE_URL`/`MODEL`, `REQUESTY_API_KEY → RECALLAI_REQUEST_KEY` fallback, identical `Speaker: text` rendering). Prompts the model for a **strict JSON array** `[{ title, assignee, dueDate }]` (injecting today's date so relative deadlines resolve to ISO dates; `assignee`/`dueDate` are `""` when unstated), then parses defensively (strips ```` ```json ```` fences / isolates the first `[…]`). **Never throws** — returns `{ items, error? }` (empty `items` + a reason on failure) so the UI can show an editable list with a note. Another sibling module (the `chat.js` precedent); `summarize.js` is untouched.

- **`preload.js` / `renderer.js` / `index.html` / `index.css`** — UI, a **two-pane notepad**: a sidebar with a status pill + a **Huddle** record button, a **search box**, a list of past meetings (newest-first, relative date, active highlight, hover-to-delete), and a collapsible **Trash** section pinned to the bottom (count badge + per-item Restore ↩ / Delete-forever ✕); and an editor pane (editable title, meta line, a three-way **Summary | Transcript | Chat** toggle, and the matching panes — a full-height `<textarea>` for the editable note `content`, a read-only transcript view, and a chat message thread + input). The preload `contextBridge` exposes `window.recall.onStatus`, `onRecordingComplete`, and eighteen request/response actions backed by `ipcMain.handle`: `sendToSlack`, `listSlackChannels`, `listSlackUsers`, `saveTranscript`, `saveRecording`, `listMeetings`, `updateMeeting`, `deleteMeeting`, the trash three — `listTrash`, `restoreMeeting`, `deleteMeetingPermanent` — `askMeeting`, `startHuddle`, `stopHuddle`, and the GetOverview four — `listGetOverviewProjects`, `createGetOverviewTask`, `sendGetOverviewTranscript`, `extractActionItems`. The renderer loads history (and the trash) on `DOMContentLoaded`, opens the newest note (always on the Summary view + Slack destination), **auto-saves summary/title edits with a 1s debounce** (`updateMeeting` with `{ content }` or `{ title }`), and wires the action buttons to the *open* meeting.

  **View scoping (Summary-only actions).** `setView` mirrors the old toggle but now also hides the `.note-actions` row and the `.send-to` block whenever the view isn't `summary` — so **Transcript shows only the transcript, Chat shows only the chat**, and the action/send controls (which act on the summary) appear only on the Summary view. The two wrapper refs (`noteActionsEl` = `.note-actions`, `sendToEl` = `.send-to`) are toggled by the `hidden` attribute.

  **Action buttons** (in `.note-actions`, icon + `.action-text` span, inline-SVG like the Slack logo): a **Recording** play link (`#video-link`, opens `videoUrl` in the browser), a **Download** button (`#save-recording-btn`, saves the MP4) — both shown only when `videoUrl` exists — and a **Transcript** button (`#save-transcript-btn`, native save). `runSave` and the `showEditor` resets call **`setActionLabel(btn, text)`**, which writes to the inner `.action-text` span (falling back to `textContent` for plain buttons) so the busy→done→idle states don't wipe the SVG icon.

  **"Send to" destination switcher.** A `.send-to` block holds a "Send to" heading + a **Slack | GetOverview** toggle (`#dest-slack-btn` / `#dest-go-btn`, reusing `.note-toggle`/`.toggle-opt`) and two panels (`#dest-slack-panel` / `#dest-go-panel`). `setDest('slack'|'getoverview')` mirrors `setView` — toggles the active button and the panels' `hidden`. **Defaults to Slack**, and `showEditor` resets to Slack per opened note (like defaulting the view to Summary). Both destination dropdowns are **global controls** (like Huddle) — selections aren't stored per-meeting.

  **Slack panel** — a Slack-themed dropdown + logo "Send" button, populated once on `DOMContentLoaded` by `loadSlackTargets` (fetches `listSlackChannels` + `listSlackUsers` in parallel, builds two `<optgroup>`s — **Channels** `#name`/🔒-private and **People** `@display-name`); a group degrades to a disabled status line on its own error so the other still works. Option values are typed (`channel:<id>` / `user:<id>`); the send handler splits this to pass `{ channel: id, isDm: kind === 'user' }` to `sendToSlack` (the **edited** `editorEl.value`).

  **GetOverview panel** — a project dropdown populated by `loadGetOverviewProjects` on `DOMContentLoaded` (and a ↻ refresh button), **grouped into `<optgroup>`s by project `status`** (the only group-able field GetOverview exposes); an **Open in GetOverview** link bound to the selected project's `url`; **Send summary** / **Send full transcript** buttons (the same `sendGetOverviewTranscript`, differing only in payload — `editorEl.value` vs. the rendered `Speaker: words` transcript — and `sourceId` suffix so the two don't collide on GetOverview's idempotency key); and **Create tasks from action items**, which calls `extractActionItems`, renders editable rows (title / assignee text / `<input type=date>` / remove), then on submit loops `createGetOverviewTask` per row into the selected project, marking each row done/error in place. A bad/unset token degrades the dropdown to a disabled "not configured"/error line.

  The **Huddle button** toggles a manual recording (idle "Huddle" → red "Stop"): a click calls `startHuddle`/`stopHuddle` and drives the status pill; it resets to idle when the finished note arrives on `recording-complete`. It's a global control (always visible). The **search box live-filters the sidebar** (`meetingMatches` — case-insensitive substring over title + note `content` + transcript text, reusing `segmentParts`); a pure in-memory filter over the loaded `meetings`, an empty query shows the full list. The **Trash section** (`renderTrash`) is loaded separately via `listTrash` into `trashedMeetings`, collapsed by default (toggling flips `trashExpanded`); deleting a note moves it into `trashedMeetings` in-memory (so the count updates without a reload) while `deleteMeeting` soft-deletes on disk, **Restore** (`restoreMeeting`) pushes it back into `meetings` and opens it, and **Delete-forever** (`purgeMeeting` → `deleteMeetingPermanent`) is the only irreversible action. The chat submit handler posts the question via `askMeeting`, shows a transient "Thinking…" bubble, renders the answer, then persists both turns via `updateMeeting({ chat })`. Node integration is off; **all** main↔renderer communication goes through the preload bridge. Native save dialogs and filesystem writes live in `main.js` (only the main process has `dialog`/`fs`).

### Critical, non-obvious constraints

These are the things that broke during development and must be preserved:

1. **Register every SDK event listener BEFORE `RecallAiSdk.init()`.** The SDK wires its detection subsystem at init time; listeners added after init never fire. `init()` is intentionally the last call in the ready handler in `main.js`.

2. **The backend uses port 3100, not 3000.** Electron Forge's webpack dev server occupies 3000. A clash leaves API routes unregistered, which surfaces as a 404 on `create_sdk_recording`. `server.start()` resolves `null` (never throws) on `EADDRINUSE` so a stale process can't crash the app.

3. **The Recall SDK must stay out of the webpack bundle.** `webpack.main.config.js` lists it in `externals`, and `forge.config.js` uses `@timfish/forge-externals-plugin` + asar `unpackDir: 'node_modules/@recallai'`. The SDK resolves its native agent binary (`agent-windows.exe`) relative to `__dirname`; bundling it breaks `__dirname` so the native agent can't launch.

4. **Transcription must be requested explicitly.** `create_sdk_recording` sends `recording_config.transcript.provider` in the POST body. With no body, Recall uploads media but never generates a transcript (`media_shortcuts.transcript` stays absent → transcript URL is `undefined`).

5. **Completion is poll-based, not webhook-based (and needs no public endpoint/ngrok).** At record-start, `create_sdk_recording` returns `{ id, upload_token }`; `main.js` keeps the `id` (the `desktop_sdk_upload` id) in a module-level `pendingUploadIds` list. On the SDK's local `recording-ended` event, `main.js` drains that list and calls `backend.processCompletedUpload(uploadId)` for each. The recording's own id is read from the polled recording object (`recording.id`), not from any event payload.

6. **Dedup is by resolved recording id.** `processCompletedUpload` adds `recording.id` to a module-level `handledRecordings` `Set` and skips if already present, so a second `recording-ended` (or a stray re-poll) for the same recording is a no-op. (Draining the whole `pendingUploadIds` list on each end is therefore safe.)

7. **The poll is long, because media lags `recording-ended` by *minutes*.** Recall uploads/processes the recording server-side after the call ends. `processCompletedUpload` polls `GET /api/v1/recording/?desktop_sdk_upload_id=<id>` ~20× with backoff (3s → capped at 15s, ~3.5 min total) until **both** `media_shortcuts.transcript.data.download_url` and `…video_mixed.data.download_url` appear (or it times out and forwards what it has). This is the outbound replacement for the old inbound webhook.

8. **The app must stay open after a call** for the poll to run (same practical requirement as the old webhook). If closed mid-processing, the pending upload id is lost (no on-disk persistence/re-poll — a possible future enhancement).

9. **The AI summary is generated inside `processCompletedUpload`** (after the transcript downloads), then `server.js` emits `{ recordingId, videoUrl, transcriptUrl, transcript, summary }` to its `setRecordingCompleteHandler`; the renderer only displays strings, so no streaming IPC channel exists.

10. **`main.js` persists before forwarding; the renderer receives a *meeting*, not the raw backend payload.** The complete handler turns the backend payload into a meeting via `buildMeeting`, `store.upsertMeeting`s it, and sends that saved meeting object on `recording-complete`. So the renderer's `onRecordingComplete` shape is the meeting (`{ id, title, date, content, summary, transcript, videoUrl, … }`), and disk + UI stay in agreement. If the write fails, it still forwards the in-memory meeting so the session isn't lost.

11. **Slack sends the edited note, not the stored summary.** The renderer passes the live `<textarea>` value (`meeting.content` as edited) to `sendToSlack`, so user refinements go out. `summarize.js` and the completion/summary flow are unchanged — the summary is just the *initial* `content`.

11b. **Slack uses the Bot Token Web API, not the old Incoming Webhook.** The webhook was single-channel; the bot token (`Bot_User_OAuth_Token`) lets the user pick any channel **or a person** per send. Two non-obvious Slack facts baked into the code: (a) the Web API returns **HTTP 200 with `{ ok:false, error }`** on logical failures (bad token, `not_in_channel`, `missing_scope`) — always check the parsed `ok`, not just `res.ok`; (b) a DM to a person ID **always renders in the recipient's sidebar under the app's name** ("Notetaker" → Messages), never as a human-to-human DM bubble — this is normal Slack behavior for bot DMs, **not** a bug. `chat.postMessage` also accepts a user ID directly, but we `conversations.open` first to resolve the DM channel robustly.

12. **Chat is independent of autosave and Slack.** The chat thread feeds neither the summary autosave nor the Slack send. It's persisted explicitly with `updateMeeting({ chat })` after each answer completes, sending the **full untruncated transcript** plus prior turns to `chat.js`. A very long meeting can therefore exceed the model's context window — that surfaces as a readable "Chat unavailable" bubble, not a crash. Old notes lacking a `chat` field default to an empty thread in the renderer (no migration).

13. **The manual Huddle is a second entry point into the *same* pipeline — not a parallel one.** `prepareDesktopAudioRecording()` returns a desktop-audio `windowId` (the manual analogue of `meeting-detected`'s `evt.window.id`); from `startRecording` onward everything is shared — the upload-id capture into `pendingUploadIds`, the `recording-ended`-triggered `processCompletedUpload` poll + dedup, `summarizeTranscript`, and the persist-then-forward `recording-complete` flow are all reused unchanged, so a huddle becomes an ordinary note indistinguishable from an auto-recorded one (and is searchable, Slack-able, etc., for free). Two non-obvious requirements: `'system-audio'` must be in `acquirePermissionsOnStartup` for desktop-audio capture, and `create_sdk_recording` must request `video_mixed_mp4` (it does) for the huddle to capture the screen. There is no manual-only code path on the backend or store.

14. **GetOverview is a second outbound integration shaped exactly like Slack — read-mostly, never-throw, optional.** It mirrors the Slack pattern across every layer (never-throw `{ ok, … }` backend helpers, thin IPC pass-throughs, `contextBridge` actions, a global dropdown). Three things about the **actual GetOverview API** constrain the feature (it is narrower than "full PM control"): (a) **there is no create-project endpoint** — projects are read-only (`GET /api/v1/projects`), so the UI lists them but cannot add one; (b) **there is no list-tasks endpoint** — only single-task get/create/update — so "see tasks" is replaced by an **Open in GetOverview** link to the project `url`; (c) **there is no members/users endpoint** — so **assignee is free text** (email or name), not a dropdown. Projects have no group field other than `status`, which is what the dropdown groups by. The transcript-submit endpoint returns a **202 `{ jobId, status }`** and processes server-side; it does **not** return the generated summary/action-items, so the app confirms acceptance and does not poll results. Action-item extraction is a **separate AI call** (`extractActionItems.js`) from the meeting summary (`summarize.js` is untouched) — the GetOverview Summary panel deliberately shows **options only**, not a duplicate of the summary (which lives in the Summary view). GetOverview/Slack selections are ephemeral UI state, never persisted on the meeting.

### Output & persistence

**Meeting notes are persisted automatically** to `meetings.json` in `userData` (`%APPDATA%/<app>/meetings.json` on Windows) — every completed recording becomes a note, and the renderer's debounced edits to the title/body **plus the chat thread** are written back through `update-meeting`. **Deletes are soft:** the sidebar 🗑 button moves a note to the Trash (stamps `deletedAt`) rather than removing it, where it can be **Restored** or **Deleted forever** (the only path that actually removes the row). This is the only thing written automatically; exporting transcripts/recordings to arbitrary locations is still **manual** via native dialogs. The editor pane has a **Summary / Transcript / Chat** toggle (opens on Summary). The action buttons and the **"Send to" Slack | GetOverview** controls below are **shown only on the Summary view** (Transcript/Chat panes show only their content — constraint via `setView`); all act on the open note:
- **Send to Slack** → posts the **edited note text** to the selected channel or person (DM) via the Slack Web API bot token (see constraints 11 / 11b).
- **Send to GetOverview** → **Send summary** (edited note) or **Send full transcript** submits to GetOverview's transcript endpoint; **Create tasks from action items** AI-extracts editable rows and creates one task per row in the selected project (see constraint 14).
- **Transcript** (icon button, was "Save transcript as…") → native save dialog (default `transcript_<id>.txt`, switchable to `.json`); `.txt` is the readable **full transcript** `Speaker: words`, `.json` is `{ recordingId, transcript }`. (Saves the transcript, not the note.)
- **Recording** (play) + **Download** → **Recording** opens the MP4's `videoUrl` in the browser; **Download** saves it via native dialog (`recording_<id>.mp4`, loaded fully into memory via `arrayBuffer`). Both shown only when a video URL exists.

The `recordings/` folder is no longer auto-populated (it's just the default the user may pick). `formatTranscript` lives in `server.js` (exported, reused by main.js for the Save-transcript path) — it's no longer used by `sendToSlack`, which now sends the note text. The renderer's Transcript view re-derives `Speaker: words` from the segments itself (`segmentParts` + `renderTranscript` in `renderer.js`, and `transcriptText` reuses it for the GetOverview full-transcript submit) using the same extraction logic as `server.js`'s `formatTranscript` and `chat.js`/`summarize.js`/`extractActionItems.js` — keep these copies consistent by hand.

## Platform notes

The Desktop SDK supports Windows 10+ (64-bit) and macOS 13+ (Apple Silicon only). This codebase is currently wired for the Windows native agent (`agent-windows.exe`); a Mac build would use a different native binary and needs macOS permissions (full-disk access for URL extraction).
