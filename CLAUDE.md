# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this app is

An Electron desktop app that uses the **Recall AI Desktop SDK** (`@recallai/desktop-sdk`) to detect meetings (Google Meet in Chrome/Edge on Windows), record them, and retrieve transcripts by **polling** the Recall API after the recording ends (no inbound webhook / public endpoint required). Detection is **opt-in, not auto-record**: when a meeting is detected the app shows a **Granola-style "Meeting Detected" popup** (a separate frameless, always-on-top window, top-right of the screen) with a **"Start Recording"** button, plus a dynamic in-app **Start/Stop Recording** button in the sidebar — the user starts the recording (either entry point), it does **not** start on its own. Recording can also be started **manually** with a **"Huddle"** button (captures the screen + desktop/mic audio without a detected meeting). When a recording completes, it generates a structured **AI summary** of the transcript, **persists it as a meeting note** to a local JSON file, and presents it in a **notepad-style UI** — a sidebar of past meetings (with a **Trash** section at the bottom: deletes are soft, recoverable, and can be restored or permanently purged) plus an editor with three toggle views: the editable **Summary** (refinable before sending out), the read-only **Transcript**, and an **AI Chat** that answers questions grounded in that meeting's transcript. From the Summary view the note can be sent to two destinations via a **"Send to" Slack | GetOverview toggle**: **Slack** (channel or DM) and **GetOverview** (the user's internal project-management tool — submit the summary/transcript, or AI-extract action items into editable rows and create tasks in a project). Built with Electron Forge + webpack.

### Releasing (GitHub Releases auto-update)

The app auto-updates from **GitHub Releases** (public repo `spread-the-rumor/DesktopNote`). Releasing is tag-driven and runs in CI — you don't build locally:

1. Bump `version` in **both** `package.json` and `package-lock.json`, commit.
2. `git tag vX.Y.Z && git push origin main && git push origin vX.Y.Z`.
3. `.github/workflows/release.yml` (Windows runner) runs `npm run publish`, which builds the Squirrel installer and uploads it to a **draft** GitHub Release (`draft: true` in `forge.config.js`).
4. Publish the draft to make it live: `gh release edit vX.Y.Z --draft=false`. Until published, `update.electron.org` won't serve it.

Installed apps poll `update.electron.org` (free, **public-repo only**) on launch + hourly (`update-electron-app` in `main.js`, gated on `app.isPackaged`), download in the background, and show a built-in restart-to-apply prompt. **Unsigned** — first install shows a one-time Windows SmartScreen "unknown publisher" warning (acceptable for the internal team; revisit before external distribution).

There is no test suite and no real linter (`npm run lint` is a placeholder echo). Verification is done by syntax-checking changed files and by running the app end to end.

### Running the full pipeline locally

The full pipeline runs entirely locally — no public reachability, ngrok, or webhook config needed:
1. `npm start` — the in-process backend logs `Backend running on http://localhost:3100`.
2. The app must stay open after a call ends (or after pressing **Stop** on a Huddle) — Recall processes the recording server-side for minutes after the call, and the app **polls** for the finished media on the SDK's `recording-ended` event (see "completion" below). The same applies to the manual Huddle button, since it uses the identical pipeline.

> **No public endpoint / ngrok required.** Completion is **poll-based, not webhook-based.** The app no longer exposes a `/webhook` route; instead `main.js` captures the `desktop_sdk_upload` `id` at record-start and, on `recording-ended`, calls `backend.processCompletedUpload(uploadId)`, which polls `GET /api/v1/recording/?desktop_sdk_upload_id=<id>` (outbound only) until the media is ready. ngrok and the Recall dashboard webhook config are no longer used.

**Configuration: in-app Settings, with `.env` as a dev fallback.** A packaged build has **no `.env`** (gitignored, never bundled), so keys are supplied at runtime via the **in-app Settings view** (gear icon, top-left) and persisted to `settings.json` in `userData` by `settingsStore.js`. In dev, `.env` still works (see `.env.example`) — `main.js` loads `settings.json` into `process.env` first, then `server.js`'s own `require('dotenv').config()` layers `.env` on top. `RECALL_API_KEY` is **required to record** but **no longer fatal**: a missing key now degrades gracefully (the backend warns and the `create_sdk_recording` route returns a clear "not configured" error) instead of the old `process.exit(1)` that silently killed the packaged app on launch. The Slack **Bot User OAuth Token** (read as `Bot_User_OAuth_Token`, with `SLACK_BOT_TOKEN`/`SLACK_BOT_USER_OAUTH_TOKEN` fallbacks) is optional — it powers the "Send to Slack" feature (channel/person dropdown + send); if unset, listing and sending return a clear "not configured" error instead of failing. Required bot scopes: `chat:write`, `channels:read`, `groups:read` (private channels), `channels:join` (auto-join public channels), `users:read` (the People list), `im:write` (open DMs). The old `SLACK_WEBHOOK_URL` is **deprecated and unused** — the webhook (single-channel) was fully replaced by the bot-token Web API. The AI summary needs a requesty.ai key, read as `REQUESTY_API_KEY` with a fallback to `RECALLAI_REQUEST_KEY` (the name it's stored under in this project's `.env`); if neither is set, summaries (and the action-item extraction that reuses the same key) degrade to a readable "unavailable" string rather than crashing. The **GetOverview** integration is optional too: base URL + token are read as `GetOverview_BASE_URL` and `GetOverview_Access_Token` (exact names; trailing slash on the URL is stripped). The token is a Personal Access Token (`go_live_…`) sent as `Authorization: Bearer <token>`. If either is unset, all GetOverview controls report a clear "not configured" state instead of failing, and the rest of the app is unaffected.

## Architecture

Three processes cooperate; data flows in a loop **renderer ← main ← in-process backend ← Recall**.

- **`main.js` (Electron main process)** — Orchestrator. On `app.whenReady` it (in order): inits `settingsStore`, loads `settings.json` and **layers the saved keys onto `process.env`**, then **lazily `require('./server')`** (the backend reads its keys into frozen constants at require-time, so this ordering is what makes the packaged app — which has no `.env` — able to reach Recall once a key is entered; `backend` is a module-level `let`, **not** a top-level require), starts the backend in-process (`backend.start(3100)`), points the history store at `app.getPath('userData')` (`store.init`), registers `setRecordingCompleteHandler` to **persist then forward** each completed meeting, registers **all** SDK event listeners, then calls `RecallAiSdk.init()` **last**. It also wires auto-update at module load (`updateElectronApp`, gated on `app.isPackaged`) — see "Distribution" above. The complete handler calls `buildMeeting(payload)` (derives the date/title, seeds the editable `content` from the AI summary, and seeds an empty `chat: []`), `store.upsertMeeting`s it, and forwards the *saved meeting* on `recording-complete`. Also hosts the history IPC handlers (`list-meetings`, `update-meeting`, `delete-meeting`) **plus the trash handlers** (`list-trash`, `restore-meeting`, `delete-meeting-permanent` → the matching `store.*`), the **settings handlers** (`get-settings`/`save-settings` → `settingsStore.*`; `save-settings` also writes the keys back onto `process.env`; plus `restart-app` → `app.relaunch()`+`app.quit()` and `get-app-version` → `app.getVersion()`), the chat handler (`ask-meeting` → `backend.askMeeting`), the recording handlers (`start-huddle` / `stop-huddle`, the detected-meeting `start-detected-recording` / `dismiss-detected-meeting`, and the generalized `stop-recording` — `stop-huddle` is an alias for it), the Slack handlers (`send-to-slack`, `list-slack-channels` → `backend.listSlackChannels`, `list-slack-users` → `backend.listSlackUsers`), the GetOverview handlers (`list-getoverview-projects`, `create-getoverview-task`, `send-getoverview-transcript`, `extract-action-items` → the matching `backend.*` — all thin pass-throughs like the Slack ones), plus `save-transcript` / `save-recording`. Recording is **opt-in**: on `meeting-detected` it does **not** record — it stashes the detected window id in a module-level `detectedWindowId`, sends `status: 'meeting-detected'` to the renderer (which shows the in-app Start button), and opens the **"Meeting Detected" popup** (`createDetectionPopup`, a separate `popup_window` BrowserWindow). The user then starts via **`start-detected-recording`** (records `detectedWindowId`, clears it, closes the popup) or skips via **`dismiss-detected-meeting`** (clears it, closes the popup). The shared `startRecordingForWindow(windowId)` helper (fetch upload token → `RecallAiSdk.startRecording` → push the upload id) backs both the detected path and the Huddle. **`start-huddle`** is the manual analogue: it calls `RecallAiSdk.prepareDesktopAudioRecording()` to get a desktop-audio `windowId` (no detected meeting needed) and calls `startRecordingForWindow`. Both paths set a module-level `activeWindowId`; the single **`stop-recording`** handler (`stop-huddle` is its alias) calls `RecallAiSdk.stopRecording({ windowId: activeWindowId })`. `acquirePermissionsOnStartup` includes `'system-audio'` (alongside accessibility/microphone/screen-capture) so desktop-audio capture is permitted. On quit it shuts the SDK down and closes the backend server.

- **`store.js` (required by main.js)** — Local meeting-history persistence: a single `meetings.json` in Electron's `userData` dir, shape `{ meetings: [ <meeting> ] }`. A meeting is `{ id (=recordingId), recordingId, title, date, content, summary, transcript, videoUrl, chat, createdAt, updatedAt }` — `content` is the editable note text (seeded from `summary`), `summary` keeps the original AI output, `chat` is the persisted Q&A thread (`[{ role:'user'|'assistant', content }]`). A trashed meeting additionally carries a `deletedAt` timestamp; restoring clears it. Exports `init / loadMeetings / loadTrash / upsertMeeting / updateMeeting / deleteMeeting / restoreMeeting / permanentlyDeleteMeeting`. **Trash is a soft-delete:** `deleteMeeting(id)` stamps `deletedAt` (it no longer removes the row), `restoreMeeting(id)` clears it, and `permanentlyDeleteMeeting(id)` is the old hard-delete (the only irreversible path). `loadMeetings` returns **live** notes only (no `deletedAt`), `loadTrash` returns **trashed** notes only (newest-trashed first); both are derived from a private `loadAll` raw read. `upsertMeeting`/`updateMeeting` also read via `loadAll`, so trashed rows survive every write. `updateMeeting(id, patch)` merges arbitrary patches, so the renderer persists summary edits, title edits, **and the chat thread** through the same path. Reads are forgiving (missing/corrupt file → `{ meetings: [] }`, never throws); writes serialize through a single in-flight promise chain (temp-file + rename) so the renderer's debounced auto-saves can't interleave/corrupt the file. No third-party dep.

- **`settingsStore.js` (required by main.js)** — The in-app replacement for `.env` in packaged builds. Persists the user's API keys to a single `settings.json` in `userData` (alongside `meetings.json`), **mirroring `store.js` exactly** (module-level `filePath` from `init(dir)`, forgiving reads → `{}`, serialized atomic temp-file+rename writes). Exports `init / loadSettings / saveSettings(patch)`. Keys are stored under the **exact** names the backend reads from `process.env` (`RECALL_API_KEY`, `RECALL_API_URL`, `Bot_User_OAuth_Token`, `REQUESTY_API_KEY`, `GetOverview_BASE_URL`, `GetOverview_Access_Token`) so `main.js` can layer them straight onto `process.env`. No third-party dep.

- **`server.js` (Express backend, required in-process by main.js)** — Bridges the app to the Recall API. One route: `POST /api/create_sdk_recording` (creates an SDK upload with `video_mixed_mp4` + a transcription provider and returns the upload token **and `id`** — reused unchanged by **both** the auto-detect and the manual Huddle paths). Completion is **poll-based, not a webhook**: it exports `processCompletedUpload(uploadId)`, which `main.js` calls on `recording-ended`. That function polls `fetchRecordingByUploadId` (`GET /api/v1/recording/?desktop_sdk_upload_id=<id>`, outbound only, ~20 attempts with capped backoff over ~3.5 min) until the recording's transcript + mixed-video media is ready, downloads the transcript, calls `summarizeTranscript` to generate the AI summary, dedups by the resolved recording id (`handledRecordings` Set), and notifies main via the registered handler with `{ recordingId, videoUrl, transcriptUrl, transcript, summary }` — it does **not** write anything to disk. (The old `POST /webhook|/webhook/recall` route and the ngrok requirement are gone.) Also exports the **Slack Web API** helpers — `sendToSlack` (posts the note text to a channel via `chat.postMessage`, or to a person as a DM via `conversations.open` then `chat.postMessage` when `isDm` is set; auto-joins public channels on `not_in_channel`), `listSlackChannels` (`conversations.list`, public + private, cursor-paginated), and `listSlackUsers` (`users.list`, real humans only — skips bots/apps/deactivated/Slackbot) — all keyed off the bot token and **never throwing** (they return `{ ok, … }` | `{ ok:false, error }`, surfacing Slack's logical errors verbatim). Also exports `formatTranscript` (reused by main.js for manual transcript saves) and `askMeeting` (thin wrapper over `chat.js`'s `answerQuestion`, returns `{ ok, answer }`). Also exports the **GetOverview Web API** helpers (Bearer-auth, base URL from `.env`, all token-guarded and **never throwing** — same `{ ok, … }` | `{ ok:false, error }` contract as Slack, surfacing GetOverview's logical errors verbatim): a generic `getoverviewApi(method, pathname, body)` wrapper, `listGetOverviewProjects` (`GET /api/v1/projects` → `{ id, name, status, url }`, sorted by name; the renderer groups by `status`), `createGetOverviewTask` (`POST /api/v1/projects/{projectId}/tasks` with `title`/`assignee`/`dueDate`/`description`, empty optionals omitted), `sendTranscriptToGetOverview` (`POST /api/v1/transcripts` or the project-scoped `/api/v1/projects/{projectId}/transcripts`; trims `transcript` to GetOverview's 700 KB limit; returns `{ ok, jobId, status }` — the 202 result is not polled back), and `extractActionItems` (thin wrapper over `extractActionItems.js`, returns `{ ok, items, note }`). Also runs standalone via `node server.js`.

- **`summarize.js` (required by server.js)** — Exports `summarizeTranscript(transcript, participants = [], { stream = true, progressCallback })`, which calls the OpenAI-compatible requesty.ai chat-completions API (`https://router.requesty.ai/v1`, model `google/gemma-4-31b-it`) to produce a fixed-structure Markdown summary (`# Participants` / `# Summary` / `# Action Items`). Accepts both the documented `[{ speaker, text, timestamp }]` shape and Recall's native segment shape (normalizes both). Supports streaming (parses the SSE stream, fires `progressCallback(currentText)` per chunk) and non-streaming modes. **Never throws** — returns a human-readable `"Summary unavailable — …"` string on any failure so the UI can show it as-is.

- **`chat.js` (required by server.js)** — Exports `answerQuestion(transcript, question, history = [])`, the per-meeting AI chat. Same requesty.ai plumbing as `summarize.js` (`BASE_URL`/`MODEL`, `REQUESTY_API_KEY → RECALLAI_REQUEST_KEY` fallback, identical `Speaker: text` transcript rendering). System prompt answers **only** from the transcript (says so when the answer isn't there); sends the **full untruncated transcript** plus the prior `history` turns for follow-ups; non-streaming. **Never throws** — returns a `"Chat unavailable — …"` string on failure. `summarize.js` is left untouched; this is a sibling module so the summary path is unaffected.

- **`extractActionItems.js` (required by server.js)** — Exports `extractActionItems(transcript, summary = '')`, used to pre-fill the "Create tasks in GetOverview" editor. Same requesty.ai plumbing as `chat.js`/`summarize.js` (`BASE_URL`/`MODEL`, `REQUESTY_API_KEY → RECALLAI_REQUEST_KEY` fallback, identical `Speaker: text` rendering). Prompts the model for a **strict JSON array** `[{ title, assignee, dueDate }]` (injecting today's date so relative deadlines resolve to ISO dates; `assignee`/`dueDate` are `""` when unstated), then parses defensively (strips ```` ```json ```` fences / isolates the first `[…]`). **Never throws** — returns `{ items, error? }` (empty `items` + a reason on failure) so the UI can show an editable list with a note. Another sibling module (the `chat.js` precedent); `summarize.js` is untouched.

- **`preload.js` / `renderer.js` / `index.html` / `index.css`** — UI, a **two-pane notepad**: a sidebar with a status pill + a dynamic **Start/Stop Recording** button (`#record-btn`, hidden until a meeting is detected — shows green "Start Recording" on `meeting-detected`, flips to red "Stop Recording" while recording, hides again when it ends; driven by the `onStatus` stream, the in-app twin of the popup) + a **Huddle** record button + a **Settings** ⚙ button (and an app-version badge next to the brand, fed by `getAppVersion`), a **search box**, a list of past meetings (newest-first, relative date, active highlight, hover-to-delete), and a collapsible **Trash** section pinned to the bottom (count badge + per-item Restore ↩ / Delete-forever ✕); and an editor pane (editable title, meta line, a three-way **Summary | Transcript | Chat** toggle, and the matching panes — a full-height `<textarea>` for the editable note `content`, a read-only transcript view, and a chat message thread + input) **plus a dedicated `#settings-view` pane** (see below). The preload `contextBridge` exposes `window.recall.onStatus`, `onRecordingComplete`, and the request/response actions backed by `ipcMain.handle`: `sendToSlack`, `listSlackChannels`, `listSlackUsers`, `saveTranscript`, `saveRecording`, `listMeetings`, `updateMeeting`, `deleteMeeting`, the trash three — `listTrash`, `restoreMeeting`, `deleteMeetingPermanent` — `askMeeting`, `startHuddle`, `stopHuddle`, `startDetectedRecording`, `stopRecording`, the GetOverview four — `listGetOverviewProjects`, `createGetOverviewTask`, `sendGetOverviewTranscript`, `extractActionItems` — and the **settings actions** `getSettings`, `saveSettings`, `restartApp`, `getAppVersion`. The renderer loads history (and the trash) on `DOMContentLoaded`, opens the newest note (always on the Summary view + Slack destination), **auto-saves summary/title edits with a 1s debounce** (`updateMeeting` with `{ content }` or `{ title }`), and wires the action buttons to the *open* meeting.

  **Settings view (the in-app key config).** The ⚙ button shows `#settings-view` in place of the note/empty-state (`showSettings` hides `#note`+`#empty-state`; `closeSettings`/Done restores the prior note or newest). It has password inputs for each key (Recall key marked **required**, the rest optional), `Save` → `saveSettings(patch)`, and `Done`. `SETTINGS_FIELDS` maps each input to its env-key name. On save, a **changed Recall key prompts a restart** (`restartApp`) because the backend's key constants are frozen at require-time; other keys apply live. On startup `refreshConfiguredState` reads the saved keys: with **no Recall key**, the status pill shows a "not configured" nudge, the Huddle button is disabled, and the empty state points users to Settings (the `permissions-granted` status is suppressed so it doesn't mask the nudge).

  **View scoping (Summary-only actions).** `setView` mirrors the old toggle but now also hides the `.note-actions` row and the `.send-to` block whenever the view isn't `summary` — so **Transcript shows only the transcript, Chat shows only the chat**, and the action/send controls (which act on the summary) appear only on the Summary view. The two wrapper refs (`noteActionsEl` = `.note-actions`, `sendToEl` = `.send-to`) are toggled by the `hidden` attribute.

  **Action buttons** (in `.note-actions`, icon + `.action-text` span, inline-SVG like the Slack logo): a **Recording** play link (`#video-link`, opens `videoUrl` in the browser), a **Download** button (`#save-recording-btn`, saves the MP4) — both shown only when `videoUrl` exists — and a **Transcript** button (`#save-transcript-btn`, native save). `runSave` and the `showEditor` resets call **`setActionLabel(btn, text)`**, which writes to the inner `.action-text` span (falling back to `textContent` for plain buttons) so the busy→done→idle states don't wipe the SVG icon.

  **"Send to" destination switcher.** A `.send-to` block holds a "Send to" heading + a **Slack | GetOverview** toggle (`#dest-slack-btn` / `#dest-go-btn`, reusing `.note-toggle`/`.toggle-opt`) and two panels (`#dest-slack-panel` / `#dest-go-panel`). `setDest('slack'|'getoverview')` mirrors `setView` — toggles the active button and the panels' `hidden`. **Defaults to Slack**, and `showEditor` resets to Slack per opened note (like defaulting the view to Summary).   **Slack panel** — a Slack-themed dropdown + logo "Send" button, populated once on `DOMContentLoaded` by `loadSlackTargets` (fetches `listSlackChannels` + `listSlackUsers` in parallel, builds two `<optgroup>`s — **Channels** `#name`/🔒-private and **People** `@display-name`); a group degrades to a disabled status line on its own error so the other still works. Option values are typed (`channel:<id>` / `user:<id>`); the send handler splits this to pass `{ channel: id, isDm: kind === 'user' }` to `sendToSlack` (the **edited** `editorEl.value`).

  **GetOverview panel** — a project dropdown populated by `loadGetOverviewProjects` on `DOMContentLoaded` (and a ↻ refresh button), **grouped into `<optgroup>`s by project `status`** (the only group-able field GetOverview exposes); an **Open in GetOverview** link bound to the selected project's `url`; **Send summary** / **Send full transcript** buttons (the same `sendGetOverviewTranscript`, differing only in payload — `editorEl.value` vs. the rendered `Speaker: words` transcript — and `sourceId` suffix so the two don't collide on GetOverview's idempotency key); and **Create tasks from action items**, which calls `extractActionItems`, renders editable rows (title / assignee text / `<input type=date>` / remove), then on submit loops `createGetOverviewTask` per row into the selected project, marking each row done/error in place. A bad/unset token degrades the dropdown to a disabled "not configured"/error line.

It's a global control (always visible). The **record button** (`#record-btn`, `setRecordButton(state)`) is the detected-meeting twin: hidden by default, it appears on the `meeting-detected` status as green "Start Recording" (→ `startDetectedRecording`), flips to red "Stop Recording" while recording (→ `stopRecording`), and hides on `recording-ended`/`meeting-closed`/`meeting-dismissed`/`recording-complete` — all driven by the `onStatus` switch (and the matching `STATUS_STATES` entries, e.g. `meeting-detected` = "Meeting detected — start recording?"). **The detection popup is a separate window** — `popup.html`/`popup.js`/`popupPreload.js`, registered as a second `popup_window` webpack entry in `forge.config.js` (yielding `POPUP_WINDOW_WEBPACK_ENTRY` + `_PRELOAD_` globals like the main window). Its preload exposes only `startRecording` (→ `start-detected-recording`), `dismiss` (→ `dismiss-detected-meeting`), and `onMeetingInfo`; `main.js`'s `createDetectionPopup` builds it frameless/transparent/`alwaysOnTop('screen-saver')`/`skipTaskbar`/top-right and sends it `meeting-info` on `did-finish-load`. The popup and the in-app button are redundant entry points to the same `start-detected-recording`. The **search box live-filters the sidebar** (`meetingMatches` — case-insensitive substring over title + note `content` + transcript text, reusing `segmentParts`); a pure in-memory filter over the loaded `meetings`, an empty query shows the full list. The **Trash section** (`renderTrash`) is loaded separately via `listTrash` into `trashedMeetings`, collapsed by default (toggling flips `trashExpanded`); deleting a note moves it into `trashedMeetings` in-memory (so the count updates without a reload) while `deleteMeeting` soft-deletes on disk, **Restore** (`restoreMeeting`) pushes it back into `meetings` and opens it, and **Delete-forever** (`purgeMeeting` → `deleteMeetingPermanent`) is the only irreversible action. The chat submit handler posts the question via `askMeeting`, shows a transient "Thinking…" bubble, renders the answer, then persists both turns via `updateMeeting({ chat })`. Node integration is off; **all** main↔renderer communication goes through the preload bridge. Native save dialogs and filesystem writes live in `main.js` (only the main process has `dialog`/`fs`).

### Critical, non-obvious constraints

These are the things that broke during development and must be preserved:

1. **Register every SDK event listener BEFORE `RecallAiSdk.init()`.** The SDK wires its detection subsystem at init time; listeners added after init never fire. `init()` is intentionally the last call in the ready handler in `main.js`.

2. **The backend uses port 3100, not 3000.** Electron Forge's webpack dev server occupies 3000. A clash leaves API routes unregistered, which surfaces as a 404 on `create_sdk_recording`. `server.start()` resolves `null` (never throws) on `EADDRINUSE` so a stale process can't crash the app.

3. **The Recall SDK must stay out of the webpack bundle.** `webpack.main.config.js` lists it in `externals`, and `forge.config.js` uses `@timfish/forge-externals-plugin` + asar `unpackDir: 'node_modules/@recallai'`. The SDK resolves its native agent binary (`agent-windows.exe`) relative to `__dirname`; bundling it breaks `__dirname` so the native agent can't launch.

4. **Transcription must be requested explicitly.** `create_sdk_recording` sends `recording_config.transcript.provider` in the POST body. With no body, Recall uploads media but never generates a transcript (`media_shortcuts.transcript` stays absent → transcript URL is `undefined`).

5. **Completion is poll-based, not webhook-based (and needs no public endpoint/ngrok).** At record-start, `create_sdk_recording` returns `{ id, upload_token }`; `main.js` keeps the `id` (the `desktop_sdk_upload` id) in a module-level `pendingUploadIds` list. On the SDK's local `recording-ended` event, `main.js` drains that list and calls `backend.processCompletedUpload(uploadId)` for each. The recording's own id is read from the polled recording object (`recording.id`), not from any event payload.

6. **Dedup is by resolved recording id.** `processCompletedUpload` adds `recording.id` to a module-level `handledRecordings` `Set` and skips if already present, so a second `recording-ended` (or a stray re-poll) for the same recording is a no-op. (Draining the whole `pendingUploadIds` list on each end is therefore safe.)

7. **The poll is long, because media lags `recording-ended` by *minutes*.** Recall uploads/processes the recording server-side after the call ends. `processCompletedUpload` polls `GET /api/v1/recording/?desktop_sdk_upload_id=<id>` ~20× with backoff (3s → capped at 15s, ~3.5 min total) until **both** `media_shortcuts.transcript.data.download_url` and `…video_mixed.data.download_url` appear (or it times out and forwards what it has). This is the outbound replacement for the old inbound webhook.

8. **The app must stay open after a call** for the poll to run (same practical requirement as the old webhook). If closed mid-processing, the pending upload id is lost (no on-disk persistence/re-poll — a possible future enhancement).

9. **The AI summary is generated inside `processCompletedUpload`** (after the transcript downloads), then `server.js` emits `{ recordingId, videoUrl, transcriptUrl, transcript, summary }` to its `setRecordingCompleteHandler`; the renderer only displays strings, so no streaming IPC channel exists.

10. **`main.js` persists before forwarding; the renderer receives a *meeting*, not the raw backend payload.** The complete handler turns the backend payload into a meeting via `buildMeeting`, `store.upsertMeeting`s it, and sends that saved meeting object on `recording-complete`. So the renderer's `onRecordingComplete` shape is the meeting (`{ id, title, date, content, summary, transcript, videoUrl, … }`), and disk + UI stay in agreement. If the write fails, it still forwards the in-memory meeting so the session isn't lost.

11. **Slack sends the edited note, not the stored summary.** The renderer passes the live `<textarea>` value (`meeting.content` as edited) to `sendToSlack`, so user refinements go out. `summarize.js` and the completion/summary flow are unchanged — the summary is just the *initial* `content`.

11b. **Slack uses the Bot Token Web API, not the old Incoming Webhook.** The webhook was single-channel; the bot token (`Bot_User_OAuth_Token`) lets the user pick any channel **or a person** per send. Two non-obvious Slack facts baked into the code: (a) the Web API returns **HTTP 200 with `{ ok:false, error }`** on logical failures (bad token, `not_in_channel`, `missing_scope`) — always check the parsed `ok`, not just `res.ok`; (b) a DM to a person ID **always renders in the recipient's sidebar under the app's name** ("Notetaker" → Messages), never as a human-to-human DM bubble — this is normal Slack behavior for bot DMs, **not** a bug. `chat.postMessage` also accepts a user ID directly, but we `conversations.open` first to resolve the DM channel robustly.

12. **Chat is independent of autosave and Slack.** The chat thread feeds neither the summary autosave nor the Slack send. It's persisted explicitly with `updateMeeting({ chat })` after each answer completes, sending the **full untruncated transcript** plus prior turns to `chat.js`. A very long meeting can therefore exceed the model's context window — that surfaces as a readable "Chat unavailable" bubble, not a crash. Old notes lacking a `chat` field default to an empty thread in the renderer (no migration).

13. **The Huddle and the detected-meeting start are two entry points into the *same* pipeline — not parallel ones.** Both go through the shared `startRecordingForWindow(windowId)` helper; the only difference is where `windowId` comes from — `prepareDesktopAudioRecording()` for a Huddle (desktop-audio, no meeting window) vs. the detected meeting's window id for `start-detected-recording`. From `startRecording` onward everything is shared — the upload-id capture into `pendingUploadIds`, the `recording-ended`-triggered `processCompletedUpload` poll + dedup, `summarizeTranscript`, and the persist-then-forward `recording-complete` flow — so either becomes an ordinary note (searchable, Slack-able, etc.) for free. Two non-obvious requirements: `'system-audio'` must be in `acquirePermissionsOnStartup` for desktop-audio capture, and `create_sdk_recording` must request `video_mixed_mp4` (it does) for the huddle to capture the screen. There is no manual-only code path on the backend or store.

14. **GetOverview is a second outbound integration shaped exactly like Slack — read-mostly, never-throw, optional.** It mirrors the Slack pattern across every layer (never-throw `{ ok, … }` backend helpers, thin IPC pass-throughs, `contextBridge` actions, a global dropdown). Three things about the **actual GetOverview API** constrain the feature (it is narrower than "full PM control"): (a) **there is no create-project endpoint** — projects are read-only (`GET /api/v1/projects`), so the UI lists them but cannot add one; (b) **there is no list-tasks endpoint** — only single-task get/create/update — so "see tasks" is replaced by an **Open in GetOverview** link to the project `url`; (c) **there is no members/users endpoint** — so **assignee is free text** (email or name), not a dropdown. Projects have no group field other than `status`, which is what the dropdown groups by. The transcript-submit endpoint returns a **202 `{ jobId, status }`** and processes server-side; it does **not** return the generated summary/action-items, so the app confirms acceptance and does not poll results. Action-item extraction is a **separate AI call** (`extractActionItems.js`) from the meeting summary (`summarize.js` is untouched) — the GetOverview Summary panel deliberately shows **options only**, not a duplicate of the summary (which lives in the Summary view). GetOverview/Slack selections are ephemeral UI state, never persisted on the meeting.

15. **Never reintroduce a top-level `process.exit()` (or synchronous throw) at module-load in `server.js` or its siblings.** A packaged app has no `.env`, so `RECALL_API_KEY` is undefined at startup; the old `process.exit(1)` killed the app silently before any window appeared (installer ran, nothing launched). Missing keys must **degrade gracefully** (warn + return a "not configured" error from the route), matching the Slack/GetOverview/AI pattern. This is the bug that motivated the in-app Settings feature.

16. **Load settings into `process.env` BEFORE requiring `server.js`.** `server.js` freezes its keys into top-level `const`s at require-time, so `main.js` must `settingsStore.loadSettings()` → write onto `process.env` → **then** `require('./server')` (lazy, inside `whenReady` — `backend` is a `let`). A consequence: a **Recall key changed at runtime won't take effect until restart** (hence the Settings "Restart now" prompt); the AI/Slack/GetOverview keys read via `process.env` at call-time, so they apply live.

17. **Auto-update needs a PUBLIC repo and a PUBLISHED (non-draft) release.** `update.electron.org` only serves public repos, and the publisher uploads as a **draft** — an unpublished draft is invisible to installed apps. Forgetting `gh release edit vX --draft=false` means "the update never shows up." Also: bump the version in **both** `package.json` and `package-lock.json` or they drift.

18. **NEVER change `name` in `package.json` — it is the Squirrel auto-update anchor.** Squirrel.Windows derives the package id from `package.json` `"name"` (e.g. `your-call-ai` → package id `your_call_ai`, asset `your_call_ai-<v>-full.nupkg`, and the install dir / registry key). An installed app only looks for updates to a package with **its own** id, so changing `name` makes every existing install silently stop seeing updates (it polls for the old id, the new release only lists the new id → "no update available", no error). This is exactly what broke the **`desktopnote` → `your-call-ai`** rename at v1.0.4: v1.0.3 users couldn't auto-update and had to **manually reinstall** the new `Your.Call.AI-*.Setup.exe` once (a deliberate one-time clean break). To rename the app's **display** without breaking updates, change only `productName` (window/installer title) and the UI brand strings — keep `name` fixed. The current anchor is **`your-call-ai`**; treat it as immutable. (After such a rename the old and new apps coexist in Add/Remove Programs, and meeting history lives in a `userData` dir keyed to the app name, so a renamed app starts with empty history — no migration.)

19. **For a detected meeting, record against the *freshest* window id — refresh `detectedWindowId` on `meeting-updated`.** Because recording is opt-in (the user may click Start seconds-to-minutes after detection), the window id captured at `meeting-detected` time can go stale: on Windows/Google Meet the usable window handle + URL often arrive *later*, via `meeting-updated` (`evt.window.id`). Recording a stale handle makes the Recall SDK **fail to attach to the meeting window and emit only "Host" speaker labels** (a documented SDK symptom — see its CHANGELOG). So `meeting-updated` overwrites `detectedWindowId` with the latest id **while a meeting is still pending** (it's cleared to `null` once recording starts, so an in-progress recording is never disturbed). This was the v1.0.5 fix for the "only Host" regression introduced by the opt-in delay. Note: a Huddle or a genuinely solo recording legitimately shows "Host" (Recall's synthetic desktop-audio participant, `id 32766`/`is_host:true`) — that is **not** this bug; only multi-person meetings should resolve to real names.

### Output & persistence

**Meeting notes are persisted automatically** to `meetings.json` in `userData` (`%APPDATA%/<app>/meetings.json` on Windows) — every completed recording becomes a note, and the renderer's debounced edits to the title/body **plus the chat thread** are written back through `update-meeting`. **Deletes are soft:** the sidebar 🗑 button moves a note to the Trash (stamps `deletedAt`) rather than removing it, where it can be **Restored** or **Deleted forever** (the only path that actually removes the row). This is the only thing written automatically; exporting transcripts/recordings to arbitrary locations is still **manual** via native dialogs. The editor pane has a **Summary / Transcript / Chat** toggle (opens on Summary). The action buttons and the **"Send to" Slack | GetOverview** controls below are **shown only on the Summary view** (Transcript/Chat panes show only their content — constraint via `setView`); all act on the open note:
- **Send to Slack** → posts the **edited note text** to the selected channel or person (DM) via the Slack Web API bot token (see constraints 11 / 11b).
- **Send to GetOverview** → **Send summary** (edited note) or **Send full transcript** submits to GetOverview's transcript endpoint; **Create tasks from action items** AI-extracts editable rows and creates one task per row in the selected project (see constraint 14).
- **Transcript** (icon button, was "Save transcript as…") → native save dialog (default `transcript_<id>.txt`, switchable to `.json`); `.txt` is the readable **full transcript** `Speaker: words`, `.json` is `{ recordingId, transcript }`. (Saves the transcript, not the note.)
- **Recording** (play) + **Download** → **Recording** opens the MP4's `videoUrl` in the browser; **Download** saves it via native dialog (`recording_<id>.mp4`, loaded fully into memory via `arrayBuffer`). Both shown only when a video URL exists.

The `recordings/` folder is no longer auto-populated (it's just the default the user may pick). `formatTranscript` lives in `server.js` (exported, reused by main.js for the Save-transcript path) — it's no longer used by `sendToSlack`, which now sends the note text. The renderer's Transcript view re-derives `Speaker: words` from the segments itself (`segmentParts` + `renderTranscript` in `renderer.js`, and `transcriptText` reuses it for the GetOverview full-transcript submit) using the same extraction logic as `server.js`'s `formatTranscript` and `chat.js`/`summarize.js`/`extractActionItems.js` — keep these copies consistent by hand.

## Platform notes

The Desktop SDK supports Windows 10+ (64-bit) and macOS 13+ (Apple Silicon only). This codebase is currently wired for the Windows native agent (`agent-windows.exe`); a Mac build would use a different native binary and needs macOS permissions (full-disk access for URL extraction).
